## Краткое описание проекта (Summary)

Проект представляет собой веб-приложение для совместного ведения списков (досок) с элементами (фильмы, сериалы, книги, игры и т.п.), где участники могут добавлять, редактировать и отмечать прогресс по элементам. Основная функциональность:

- Авторизация пользователей (через Convex Auth).
- Создание досок и приглашение других пользователей по email.
- На каждой доске участники добавляют элементы в свои персональные списки.
- Общий список формируется путём циклического чередования элементов всех участников (по принципу ABAB, ABCABC и т.д.).
- Каждый элемент содержит: название, общее количество эпизодов/глав, текущий эпизод, необязательную ссылку. Прогресс вычисляется автоматически.
- Участники могут менять порядок своих элементов (только в своём списке), но редактировать и удалять могут любые элементы.
- Ведётся история действий с элементами на доске.
- Ролевая модель: создатель (полный доступ, включая удаление доски), модератор (может удалять участников), участник (базовые действия со списками).
- Интерфейс на React с использованием библиотеки компонентов Mantine, роутинг через React Router, бэкенд и база данных на Convex, деплой на GitHub Pages.

## Детальный план разработки

### 1. Проектирование и подготовка

**1.1. Анализ и уточнение требований**

- Определить точную модель прав: любой участник может редактировать/удалять любой элемент (кроме изменения владельца), менять порядок только своих.
- Уточнить механизм приглашений: добавление по email с поиском существующего пользователя; если не найден – предложить отправить приглашение (в базовой версии просто сообщать об ошибке).
- Определить структуру истории: записывать события (создание, обновление полей, удаление, отметка просмотренным) с меткой времени.

**1.2. Модели данных (Convex schema)**

- `users`: автоматическая таблица от Convex Auth, дополнительно храним `email`, `name` (получаем из провайдера или запрашиваем при регистрации).
- `boards`: `_id`, `name`, `createdBy` (userId), `createdAt`, `isActive` (для мягкого удаления).
- `boardMembers`: `_id`, `boardId`, `userId`, `role` ('creator', 'moderator', 'member'), `joinedAt`.
- `items`: `_id`, `boardId`, `userId` (владелец), `title`, `totalEpisodes` (number), `currentEpisode` (number), `url` (string, опционально), `order` (number, для сортировки внутри своего списка), `createdAt`, `updatedAt`, `isHidden` (флаг для скрытия при удалении пользователя из доски).
- `history`: `_id`, `boardId`, `userId` (кто совершил действие), `itemId` (если относится к элементу), `action` ('create', 'update', 'delete', 'complete', 'uncomplete'), `details` (JSON с изменениями), `timestamp`.

Индексы: по `boardId`, `userId`, `boardId+order`, `boardId+userId+order`.

**1.3. Настройка проекта**

- Инициализация Vite + React (шаблон `react`).
- Установка зависимостей: `@mantine/core`, `@mantine/hooks`, `@mantine/notifications` (для уведомлений), `react-router-dom`, `convex`, `convex/react`, `@convex-dev/auth` (если используется встроенная аутентификация), `prettier`, `eslint`, `eslint-plugin-import`, `eslint-plugin-react`, `eslint-plugin-react-hooks`.
- Настройка алиасов `@` → `src` в `vite.config.js` и `jsconfig.json` для поддержки абсолютных импортов.
- Настройка ESLint и Prettier с правилами для автоформатирования и сортировки импортов (`import/order`).
- Добавление Husky и lint-staged для автоматического форматирования при коммите (опционально).
- Настройка Mantine: импорт стилей, установка темы (при необходимости).

### 4. Функциональность досок

**4.1. Список досок (главная страница)**

- Вывод досок, где пользователь является участником (с ролью).
- Кнопка создания новой доски (модальное окно с полем названия).
- При создании создатель автоматически добавляется в `boardMembers` с ролью 'creator'.

**4.2. Просмотр доски (страница доски)**

- Заголовок доски, кнопки управления (для создателя/модератора).
- Вкладки: "Общий список", "Мои элементы", "Участники", "История".
- Отображение общего списка с чередованием (см. п.6).
- Отображение своих элементов с возможностью изменения порядка (drag-n-drop).

**4.3. Управление участниками**

- Для создателя/модератора: список участников с ролями.
- Возможность добавить нового участника по email (поиск пользователя в БД).
- Изменение роли (создатель может назначать модераторов, модераторы могут только удалять участников).
- Удаление участника: при удалении все его элементы получают `isHidden = true` (скрываются из общего списка), но остаются в истории.

**4.4. Удаление доски**

- Только для создателя: кнопка удаления (мягкое удаление – установка флага `isActive = false`, либо полное удаление каскадно).

### 5. Функциональность элементов

**5.1. Создание элемента**

- Форма (название, общее кол-во, текущее кол-во, ссылка) – доступна в разделе "Мои элементы".
- После создания элемент привязывается к текущему пользователю и доске, получает `order` = максимальный `order` пользователя + 1.

**5.2. Редактирование элемента**

- Любой участник может открыть модальное окно с полями элемента (кроме `userId`) и изменить их.
- При изменении обновляется `updatedAt`, в историю добавляется запись с изменениями (можно хранить только факт изменения, без деталей для простоты).

**5.3. Отметка "просмотрено"**

- Чекбокс или кнопка, которая устанавливает `currentEpisode = totalEpisodes` (или отдельный флаг). Прогресс вычисляется как `currentEpisode / totalEpisodes * 100%`.

**5.4. Удаление элемента**

- Любой участник может удалить элемент (кнопка в карточке).
- При удалении элемент помечается как `isHidden = true` (скрывается из списков) и записывается в историю. Можно также полностью удалять, но тогда теряется история – по ТЗ лучше скрывать.

**5.5. Изменение порядка своих элементов**

- В разделе "Мои элементы" реализовать drag-n-drop с обновлением `order` для всех элементов пользователя в доске.
- После изменения порядка вызывать мутацию Convex, которая пересчитывает порядок (например, присваивает новые значения в зависимости от позиции).

### 6. Алгоритм чередования (общий список)

**6.1. Подход**

- Получить всех активных участников доски (не удалённых).
- Для каждого участника получить его элементы с `isHidden = false`, отсортированные по `order`.
- Выполнить циклическое слияние массивов: пока есть хотя бы один непустой массив, брать по одному элементу из каждого массива по очереди, пропуская пустые.
- Если у какого-то участника элементы кончились, он пропускается в следующих циклах.

**6.2. Реализация**

- Функция `getMergedBoardItems(boardId)` в Convex query, которая возвращает объединённый массив с метаданными (имя пользователя для каждого элемента). Либо делать слияние на клиенте – для простоты и гибкости, но тогда нужно загрузить все элементы. Приемлемо, если элементов не сотни.
- При изменении элементов (добавление, удаление, изменение порядка, скрытие) общий список пересчитывается заново.

### 7. История

**7.1. Запись событий**

- При каждом действии с элементом (создание, обновление, удаление, отметка просмотренным) вызывать мутацию `logHistory`, передавая нужные данные.
- В истории хранить: `boardId`, `userId`, `itemId` (если применимо), `action`, `details` (JSON), `timestamp`.

**7.2. Отображение истории**

- Вкладка "История" на странице доски – список событий с фильтром по пользователям/элементам, отсортированный по времени (новые сверху).
- Использовать Mantine Timeline или простой список.

### 9. Интеграция с Convex

**9.1. Настройка Convex**

- Установить Convex CLI, инициализировать проект.
- Определить schema и функции (queries/mutations) согласно моделям.
- Настроить аутентификацию через `@convex-dev/auth`.
- Создать функции:
  - `boards: list, create, get, update, remove`
  - `boardMembers: add, updateRole, remove`
  - `items: listByBoard, listByUser, create, update, delete, reorder`
  - `history: listByBoard`
  - `users: searchByEmail` (для приглашений)

**9.2. Безопасность и валидация**

- В каждой мутации проверять права: наличие пользователя в доске, соответствие роли для административных действий.
- Использовать валидацию данных на стороне Convex (например, через `v`).

### 10. Деплой на GitHub Pages

**10.1. Настройка Vite**

- Установить `gh-pages`.
- В `vite.config.js` указать `base: '/<repository-name>/'`.
- В `package.json` добавить скрипты: `"predeploy": "npm run build", "deploy": "gh-pages -d dist"`.

**10.2. Обработка роутинга**

- Использовать `HashRouter` для избежания проблем с History API на GitHub Pages.
- Если использовать `BrowserRouter`, создать `404.html` с редиректом на `index.html` (стандартный спа-фолбек).

**10.3. Переменные окружения**

- Для Convex URL и других ключей использовать `.env` файлы (но не коммитить). На GitHub Pages переменные окружения подставляются на этапе сборки.
